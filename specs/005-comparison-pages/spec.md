# Feature Specification: 比較結果ページの本実装

**Feature Branch**: `005-comparison-pages`

**Created**: 2026-07-05

**Status**: Draft

**Input**: User description: "フェーズ3のダミー表示を、実際の比較データ・UI
（比較表、比較テーマ一覧、フィルタなど）として磨き込む"

## User Scenarios & Testing

### User Story 1 - 比較テーマ一覧ページの本実装 (Priority: P1)

ユーザーがトップページにアクセスすると、公開済みの比較テーマが
視覚的に整理されたカード一覧で表示される。
テーマごとに含まれる比較軸の数と対象プロバイダーが一目で分かる。

**Why this priority**: サイトの入口であり、
ユーザーが目的の比較情報を見つけるための導線として最も重要。

**Independent Test**: トップページにアクセスし、
公開済みテーマがカード形式で表示され、各カードにテーマ名・
比較軸数・対象プロバイダーアイコンが含まれることを確認する。

**Acceptance Scenarios**:

1. **Given** 公開済みテーマが複数存在する,
   **When** ユーザーがトップページにアクセスする,
   **Then** テーマがカード形式で一覧表示され、
   各カードにテーマ名・説明文・比較軸数・対象プロバイダーが表示される
2. **Given** テーマ一覧が表示されている,
   **When** ユーザーがテーマカードをクリックする,
   **Then** そのテーマの比較軸一覧ページに遷移する
3. **Given** 公開済みテーマが存在しない,
   **When** ユーザーがトップページにアクセスする,
   **Then** 「比較データはまだありません」のメッセージが表示される
4. **Given** テーマが 10 件以上存在する,
   **When** ユーザーがページを表示する,
   **Then** ページネーションまたは無限スクロールで追加テーマを閲覧できる

---

### User Story 2 - 比較結果テーブルページ (Priority: P1)

ユーザーが特定テーマ・軸の比較ページにアクセスすると、
プロバイダー間の比較結果が見やすいテーブル形式で表示される。
各プロバイダーの要約・詳細・参照元リンクが確認できる。

**Why this priority**: プロダクトの中核価値。
ユーザーはこのテーブルを見て技術選定の判断材料を得る。

**Independent Test**: `/comparisons/{themeId}/{axisId}` にアクセスし、
プロバイダー比較テーブルが表示され、公式ドキュメントへのリンクが
クリック可能であることを確認する。

**Acceptance Scenarios**:

1. **Given** 比較結果データが存在する,
   **When** ユーザーが比較ページにアクセスする,
   **Then** プロバイダーが列として並ぶ比較テーブルが SSR で表示される
2. **Given** 比較テーブルが表示されている,
   **When** テーブルの内容を確認する,
   **Then** 各プロバイダーのサービス名・要約・詳細・参照元 URL が
   正しく表示される
3. **Given** 比較テーブルが表示されている,
   **When** 参照元リンクをクリックする,
   **Then** 公式ドキュメントが新しいタブで開く
4. **Given** 比較結果に「情報取得不可」のプロバイダーが含まれる,
   **When** テーブルを表示する,
   **Then** 該当プロバイダーに「情報取得不可」の旨が明示される
5. **Given** 存在しないテーマ・軸の URL にアクセスする,
   **When** ページが読み込まれる,
   **Then** 404 ページが表示される

---

### User Story 3 - テーマ別比較軸一覧ページ (Priority: P2)

ユーザーがテーマ詳細ページにアクセスすると、
そのテーマに含まれる全ての比較軸が一覧表示される。
各軸をクリックすると比較結果テーブルに遷移できる。

**Why this priority**: テーマと比較結果テーブルの間の導線として必要。
テーマに複数の軸がある場合にユーザーが選択できる画面。

**Independent Test**: `/comparisons/{themeId}` にアクセスし、
そのテーマに属する比較軸の一覧が表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** テーマに複数の比較軸が存在する,
   **When** ユーザーがテーマ詳細ページにアクセスする,
   **Then** 比較軸がリスト形式で表示される
2. **Given** 比較軸一覧が表示されている,
   **When** ユーザーが軸をクリックする,
   **Then** 対応する比較結果テーブルページに遷移する
3. **Given** テーマに比較軸が 1 つのみ存在する,
   **When** ユーザーがテーマ詳細ページにアクセスする,
   **Then** 比較結果テーブルページに直接リダイレクトされる

---

### User Story 4 - プロバイダーフィルタ (Priority: P3)

ユーザーが比較結果テーブルで特定のプロバイダーだけを
表示したい場合、フィルタで絞り込むことができる。

**Why this priority**: 5 社すべてではなく 2〜3 社だけを
比較したいケースに対応する利便性機能。

**Independent Test**: 比較テーブルページでプロバイダーフィルタを操作し、
選択したプロバイダーのみ表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** 5 社のプロバイダーが比較テーブルに表示されている,
   **When** ユーザーが AWS と GCP のみをフィルタ選択する,
   **Then** AWS と GCP の列のみが表示される
2. **Given** フィルタが適用されている,
   **When** ユーザーがフィルタをリセットする,
   **Then** 全プロバイダーが再表示される
3. **Given** フィルタが適用されている,
   **When** ページを共有する（URL をコピーする）,
   **Then** URL にフィルタ状態がクエリパラメータとして含まれ、
   共有先でも同じフィルタ状態で表示される

---

### Edge Cases

- 比較結果の JSON が破損している場合のエラー表示
- 極端に長い summary/details テキストの折り返し・省略表示
- プロバイダー数が 2 社の場合と 5 社の場合のレイアウト差異
- モバイル端末での比較テーブルの横スクロール対応
- DynamoDB/S3 への接続タイムアウト時のエラーページ表示
- 比較日時（comparedAt）が古い場合の「情報が古い可能性」の表示

## Requirements

### Functional Requirements

- **FR-001**: トップページは公開済みテーマを DynamoDB から取得し、
  SSR でカード一覧として表示しなければならない（MUST）。
  テーマカードにはテーマ表示名（title）・説明文（description）・
  比較軸数・対象プロバイダーを表示する
- **FR-002**: 比較結果ページは S3 から result.json を取得し、
  SSR でプロバイダー比較テーブルとして表示しなければならない（MUST）
- **FR-003**: 比較テーブルはプロバイダーを列（横方向）、属性を行（縦方向）として配置し、
  各セルにサービス名・要約・詳細・参照元 URL を表示しなければならない（MUST）。
  details は summary の下に常時展開で全文表示する。
  モバイルでは横スクロールで全列を閲覧可能にする
- **FR-004**: 参照元 URL はクリック可能なリンクとして表示し、
  新しいタブで開かなければならない（MUST）
- **FR-005**: テーマ詳細ページはテーマに属する比較軸の一覧を
  表示しなければならない（MUST）
- **FR-006**: テーマに比較軸が 1 つのみの場合、
  比較結果テーブルページに直接リダイレクトしなければならない（MUST）
- **FR-007**: 存在しないテーマ・軸の URL にアクセスした場合、
  404 ページを表示しなければならない（MUST）
- **FR-008**: プロバイダーフィルタ機能を提供し、
  選択したプロバイダーのみ表示できなければならない（MUST）
- **FR-009**: フィルタ状態は URL のクエリパラメータに反映し、
  URL 共有で同じ表示を再現できなければならない（MUST）
- **FR-010**: モバイル端末での比較テーブルは横スクロール可能に
  しなければならない（MUST）
- **FR-011**: 比較日時（comparedAt）を比較結果ページに表示し、
  comparedAt が現在日時から 90 日以上前の場合は
  「情報が古い可能性があります」と注意書きを表示しなければならない（MUST）
- **FR-012**: 各ページに適切な SEO メタデータ（title、description）を
  設定しなければならない（MUST）
- **FR-013**: データ取得に失敗した場合、ユーザーフレンドリーな
  エラーページを表示しなければならない（MUST）
- **FR-014**: テーマ一覧は 1 ページあたり最大 12 件を表示し、
  13 件以上の場合はページネーションを提供しなければならない（MUST）

### Key Entities

- **テーマカード**: テーマ名・説明文・比較軸数・対象プロバイダーを表示する UI 要素
- **比較テーブル**: プロバイダーを列、属性を行とする比較表
- **プロバイダーフィルタ**: 表示プロバイダーを絞り込む UI コントロール
- **比較軸一覧**: テーマに属する軸のリスト表示

## Success Criteria

### Measurable Outcomes

- **SC-001**: ユーザーがトップページから目的の比較結果に
  3 クリック以内で到達できる
- **SC-002**: 比較結果ページの初回表示が 2 秒以内に完了する
- **SC-003**: モバイル端末（幅 375px）でも全情報が閲覧可能である
- **SC-004**: 全ページが Lighthouse アクセシビリティスコア 90 以上を達成する
- **SC-005**: URL 共有によるフィルタ状態の再現が 100% 正確である

## Assumptions

- 003-frontend（歩くスケルトン）が完了しており、
  Next.js + Amplify Hosting のデプロイパイプラインが稼働している
- DynamoDB と S3 に比較結果データが存在する
  （004-ai-agent で生成されたデータ、または手動投入データ）
- 初期フェーズは社内向け（100 人程度）のため、
  ISR（1 時間再生成）で十分な鮮度を確保する。
  専用 CDN の導入は将来フェーズとする
- 認証は不要（社内ネットワークからのアクセスを想定）
- デザインシステムやブランドガイドラインは未策定のため、
  Tailwind CSS のデフォルトスタイルをベースに構築する
- 多言語対応は不要（日本語のみ）

## Clarifications

### Session 2026-07-05

- Q: 比較テーブルのレイアウト方向は？
  → A: プロバイダーを列（横）、属性（サービス名・要約・詳細・参照元）を行（縦）に配置。
  モバイルでは横スクロールで対応
- Q: 比較結果データの SSR キャッシュ・再生成方針は？
  → A: ISR（Incremental Static Regeneration）で 1 時間ごとに再生成。
  常に最新である必要はなく、月数件の更新頻度に対して 1 時間は十分な鮮度
- Q: テーマカードの「説明文」のデータソースは？
  → A: DynamoDB に `title`（テーマ表示名）と `description`（説明文）
  フィールドを追加する。Issue テンプレートの入力値から同期時に設定する
- Q: 比較テーブルの details フィールドの表示方式は？
  → A: 常時展開。summary の下に details を全文表示する
- Q: FR-011 の「90日以上前」の判定基準フィールドは？
  → A: `comparedAt`（result.json 内の比較実行日時）を使用する
